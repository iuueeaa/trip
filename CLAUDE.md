## 応答ルール(最優先)

### 基本原則
- 結論を先に、1〜3行で返答
- 思考プロセス・背景説明は出力しない
- ユーザーが聞いていない観点の提案をしない
- 作業の実況中継をしない

### 禁止する出力
- 「まず○○、次に△△…」の手順実況
- 「以下の点を検討しました」の列挙
- 作業完了後の長文サマリ
- 「補足」「ちなみに」等の追加情報
- 思考プロセスの露出

### 応答フォーマット

#### タスク完了時
```
✓ [何をしたか、1行]
  ファイル: [path](必要なら)
  次: [次のアクション、あれば]
```

#### 確認が必要な時
```
? [確認事項、1行]
  A: [選択肢]
  B: [選択肢]
```

#### 問題発生時
```
✗ [問題、1行]
  原因: [1行]
  対応: [1行]
```

### 長文を許容する例外
以下のみ詳細出力OK:
- ユーザーが「詳しく」「詳細を」と明示要求
- 原因分析レポート(docs/tickets/ 記録用)
- 複数の根本解決案の比較

### 思考の扱い
単純タスク(ファイル作成・追記・置換・整形)では、
思考プロセスをチャットに表示しない。
結論のみ出力する。

## AI役割分担（必ず守る）

| 担当   | やること                                                  | やらないこと     |
| ------ | --------------------------------------------------------- | ---------------- |
| Opus   | 設計・判断・.md編集・コメント追記・コンポーネント一覧更新 | 実装コードを書く |
| Sonnet | 新規実装                                                  | 設計判断         |
| Codex  | 修正・差分対応                                            | 新規実装         |

### Opusへの注意

- 実装コード（.php, .scss, .js）は絶対に書かない
- **例外: `TODO :` / `@fix` / `@question` コメントの追記・更新のみ許可**
- 実装が必要な場合は「指示書」を作成して終わる
- 「ついでに直しておきました」は禁止

### Sonnetへの注意

- 指示書に従って実装する
- 指示にないことは実装しない
- 不明点は `@question` コメントまたは報告で伝える
- **修正時は `value/page/*.php` のデータ定義や既存の parts/blocks との整合性を必ず確認すること**
- **修正後は必ず構文チェックを実行し、エラーがある場合は修正してから報告すること**
  - PHP: `php -l ファイル名`
  - JS: `node --check ファイル名`

### Codexへの注意

- 指示された範囲のみ修正する
- 「ついでに」のリファクタリング禁止
- 修正後は差分を報告する
- **修正時は `value/page/*.php` のデータ定義や既存の parts/blocks との整合性を必ず確認すること**
- **修正後は必ず構文チェックを実行し、エラーがある場合は修正してから報告すること**
  - PHP: `php -l ファイル名`
  - JS: `node --check ファイル名`

## 禁止事項（絶対に守る）

### 勝手にやらないこと

- 「ついでに」の修正・改善
- 指示されていないファイルの編集
- 指示にない機能の実装
- 既存コードの「リファクタリング」「改善」
- コメントの勝手な削除・追加
- 変数名・関数名の勝手な変更

### 確認なしでやらないこと

- CLAUDE.md の変更（ユーザーの許可を得てから行う）
- 複数ファイルにまたがる変更
- 新しいファイルの作成
- 既存の関数・メソッドの削除
- **AI による直接 commit**（詳細は「## Git 運用」参照）

### 同じ間違いを防ぐルール

- 過去に指摘された問題は繰り返さない（`docs/checklist-antipatterns.md` を参照）
- 不明点は実装前に質問する
- 「たぶんこうだろう」で補完しない（詳細は「## AI判断バイアス対策 > 不確実性の表明」参照）
- 指示にないことは実装しない
- **エラーだけを見て修正しない。`value/page/*.php` のデータ定義や既存の parts/blocks との整合性を必ず確認する**（原因分析の詳細は「## AI判断バイアス対策 > 安易な提案の禁止」参照）

### ソース修正のルール

- 推測でコードを修正しない。修正前に必ずソースコードとCSS構造を確認する
- HTML構造を変更する場合、対応するSCSSのセレクタ（親子関係・直接子セレクタ > ）を必ず確認する
- 既存のHTML/CSS構造が壊れないことを確認してから修正指示を出す
- **対症療法の禁止**（根本原因分析を経ない修正提案は禁止。詳細は「## AI判断バイアス対策 > 対症療法の禁止」参照）

### 参照整合性チェック(修正時の必須確認)

修正・実装後、プロジェクト固有のチェックリストを必ず確認する。**1つでも不一致があれば修正を完了としない。**

具体的なチェック項目はプロジェクト固有ルール参照:
- WordPress: docs/project-wordpress.md「## 参照整合性チェック」
- Shopify: docs/project-shopify.md(該当セクション)
- System: docs/project-system.md(該当セクション)

## 物理確認の必須化

### 対象

ファイル参照を含む処理のレビュー・修正・診断時:

- 画像・メディアファイル
- include/require/import
- アセット(CSS/JS/フォント)
- アップロードファイル
- 翻訳・設定ファイル

### 必須プロセス

コードロジックを評価する前に以下を確認:

1. 参照ファイルが実在するか
2. パスが正しく解決されるか
3. 拡張子・大文字小文字の一致
4. ファイルパーミッション

### 症状ベース診断

症状から原因を特定する場合、`docs/checklist-diagnosis.md` のチェックリストを上から順に実施。

### 禁止

- ファイル実在確認なしに「コードは正常」と判定すること
- 物理確認をスキップしてロジック修正に着手すること

## AI判断バイアス対策

### 不確実性の表明（許可と推奨）

以下を明示的に**許可・推奨**する:

- 「分からない」「自信がない」の表明
- 推測での回答より確認を求めること
- 複数解釈が可能な場合、解釈を列挙して選択を促すこと
- 「公式ドキュメントで確認すべき」の提案

**禁止される表現**:

- 不確実な事項の断言
- 「おそらく」「たぶん」を省略した断定調
- 確証のない関数名・API 名・hook 名の使用

**望ましい例**:

- ✗ 「wp_xxx 関数を使えば取得できます」
- ✓ 「wp_xxx という関数があった記憶がありますが、実在と引数仕様を公式ドキュメントで確認してください」
- ✗ 「これで動くはずです」
- ✓ 「この実装で動くと考えますが、特に〇〇のケースはローカルで動作確認をお願いします」

### AI 出力の検証

以下が含まれる場合、**実装前に検証**:

- 知らない関数・メソッド呼び出し
- 「標準装備されているはず」の主張
- 「〇〇 hook が発火する」の主張（WordPress）
- 新規ライブラリの使用
- バージョン依存の機能

**検証方法**:

1. **AI 間クロスチェック** — Sonnet 出力 → Codex 検証、逆も同様
2. **公式ドキュメント確認** — 関数・hook の実在とシグネチャを公式で確認
3. **ローカル動作確認** — ローカル環境で実行して期待通り動くか

**検証スキップが許される場合**:

- プロジェクト内で過去に動作実績がある関数・パターン
- 標準ライブラリの基本機能
- Opus が既に公式ドキュメントを参照したと明記している場合

### 安易な提案の禁止（原因分析の必須化）

修正提案前に**必ず以下を実施**:

- **Step 1: 症状の観察** — エラーメッセージ・症状を正確に記録、再現条件を特定
- **Step 2: 現状コード調査** — 該当コードの意図（コメント・コミット履歴から）、周辺コードとの関係、同じパターンの横展開調査
- **Step 3: 仕様との照合** — `docs/` の該当仕様を参照（存在する場合）、仕様書未記載なら CONFIRM として分離、仕様と現状のギャップを特定
- **Step 4: 根本原因の特定** — 症状 → 直接原因 → 根本原因を階層的に追う
- **Step 4.5: 仮説の実装検証(必須)** — 根本原因の仮説を立てたら、修正前に必ず実装で検証する
  - 仮説に関連するコードを実際に読む(grep / Read)
  - 仮説が正しいかを実装レベルで確認
  - 仮説が外れていた場合は Step 1 に戻る
  - **「たぶんこれが原因」で修正に進むことを禁止**
  - 検証結果を docs/tickets/ に記録(「仮説」と「検証結果」を分けて書く)
- **Step 5: 原因分析レポートを docs/tickets/ に記録** — 分析完了まで修正提案を出さない

### 仮説検証の禁止事項

- 仮説のまま修正コードを提示すること
- 「念のためこう直しておきます」という予防的修正
- 複数の仮説を一度に修正で潰そうとすること(1仮説ずつ検証)
- ユーザーに「これで直ると思います」と伝えること(検証済みなら「直ります」、未検証なら「検証が必要」)

### 対症療法の禁止

**原則禁止**:

- 原因不明のまま `try-catch` で覆い隠す
- 原因不明のまま `NULL` チェックを追加
- 症状だけ消す修正

**対症療法が必要な場合**:

1. 根本原因分析を先に完了
2. 暫定対応であることを明示
3. 根本解決タスクを別 FIX として登録
4. コードに `@temporary` コメントと期限を残す

例:

```php
// @temporary FIX-023 の暫定対応 (期限: 2026-05-01)
// 根本解決は FIX-024 で対応予定
if ($user === null) { return; }
```

### 提案選択肢のルール

- 本質的に異なる方向性がある場合のみ複数提示
- 同じ方向性の微差を複数提示しない
- 選択肢の「数」より「質」を優先

**良い例**: A 案「仕様を変更する方向」/ B 案「実装を仕様に合わせる方向」

**悪い例**: A 案「NULL チェック追加」/ B 案「try-catch」/ C 案「デフォルト値」（全部対症療法で方向性が同じ）

### 工数への言及制限

**原則禁止**:

- 「すぐ直せます」「簡単です」
- 「工数がかかるので A 案を推奨」
- 「最小限の変更で済みます」

判断基準は「本質的な修正か」。工数での誘導は禁止。

**許される場面**:

- ユーザーが明示的に工数を質問した場合
- 複数の根本解決案の比較として

### レビュー出力量の制限

レビュー結果が長文化して読み飛ばしを誘発するのを防ぐため、以下を遵守:

#### 件数サマリ先出し
```
レビュー完了:
- 致命的: N件
- 要修正: N件
- 軽微: N件
- CONFIRM: N件
```

#### 致命的のみチャット出力
1指摘 = 1行形式で:
```
[致命的] ファイル:行 - 問題(20字以内) → 対策(20字以内)
```

要修正・軽微は docs/tickets/ または review-details.md に記録し、チャットには出さない。

致命的がゼロなら「致命的な問題なし」と1行で返答。

### 横展開チェック必須化

指摘を行う際は、必ず以下も実施:

1. **類似パターンの横展開検索**
   - 同種の処理が他のファイル・関数にないか grep 等で確認
   - 例: NULL未処理を1箇所で指摘したら、プロジェクト全体で同じパターンを検索

2. **影響範囲の報告**
   - 指摘した関数・メソッドの呼び出し元を列挙
   - 変更した場合の波及範囲を明示

横展開検索をスキップしてはならない。

### ユーザー指示の仕様照合(必須)

ユーザーからの指示が、既存の仕様(docs/)と矛盾する場合がある。
AI(Opus/Sonnet/Codex)は指示をそのまま実行せず、必ず仕様と照合する。

#### Step 1: 指示受領直後の矛盾チェック

実装着手前に、指示内容を docs/ の関連仕様と照合する。

矛盾の例:
- 指示「全件取得して表示」 vs 仕様「ページネーション必須(20件ずつ)」
- 指示「このボタンを削除」 vs 仕様「このボタンは権限Aユーザーに必須」
- 指示「赤色に変更」 vs 仕様「ブランドカラーは青」

矛盾を発見した場合、実装に着手せず以下の形式で確認:

```markdown
## 仕様矛盾の確認

### ご指示
(ユーザーの指示内容)

### 関連する既存仕様
- docs/xxx.md(該当箇所): (仕様内容)

### 矛盾点
(指示と仕様のどこが食い違うか)

### 確認事項
A. 仕様を変更する(docs/ を更新してから実装)
B. 指示を仕様に合わせる(指示の方を見直す)
C. 例外として今回のみ仕様外実装(理由を明記)

どれで進めますか?
```

ユーザー回答後に実装着手。

#### Step 2: 実装中の都度確認

実装中に「これ仕様と合ってるかな?」と感じた箇所は、その時点で止めて確認する。

```markdown
## 実装中の仕様確認

### 該当箇所
file.php:L42 で xxx を実装中

### 気になる点
仕様書では「○○」だが、指示では「△△」と読める

### 確認
A: 仕様通り(○○)で実装
B: 指示通り(△△)で実装
C: その他
```

「たぶん指示の方が新しい」「たぶん仕様の方が正しい」で勝手に判断することを禁止。

#### 禁止事項

- ユーザー指示を仕様と照合せずに実行すること
- 矛盾に気付いていながら指摘せず実装すること
- 「指示優先 / 仕様優先」を AI が独断で決めること

#### 例外

明らかに仕様外と分かる修正(タイポ修正・コメント追加等)は照合不要。
ただし「明らか」かどうかの判断に迷ったら確認する。

## 作業の進め方

### 基本ルール

- **1回の指示で1つのことだけやる**
- 複数タスクを同時にやらない
- 終わったら報告して次の指示を待つ
- 勝手に次の作業に進まない

### やっていいこと

- 指示された範囲の修正のみ
- 質問・確認
- エラーの報告
- 禁止事項は「禁止事項」セクション参照

## 開発フロー

### 基本順序(画面駆動)

不確実性を減らすため、以下の順で進める:

1. **画面モック作成** — 静的HTML/PHPで画面イメージを作る
   → クライアント確認 / 動作イメージの確定
2. **データ枠定義** — 画面から逆算して `value/page/*.php` のデータ構造を決める
   → 必要十分なデータ項目の確定
3. **function実装** — データ構造に合わせて処理ロジックを実装
   → 仕様確定後に実装するため手戻り少

### 各段階のチェックポイント(必須)

| 段階 | 確認内容 | 確認者 |
| --- | --- | --- |
| 画面モック | 見た目・操作感の合意 | クライアント / ユーザー |
| データ枠 | キー名・型・構造の妥当性 | Opus + ユーザー |
| function | 処理ロジック・エッジケース | Opus + 動作確認 |

各チェックポイントで CONFIRM 事項を出し切ってから次段階へ進む。
チェックポイントをスキップして次段階に進むことを禁止。

### 変動する仕様への対応

クライアントレビュー / 動作確認時の仕様変更は構造上避けられない。
変更が発生したら:

1. 影響範囲を Opus が分析(影響範囲マップ作成)
2. 関連 docs/ を更新
3. 必要なら CONFIRM-xxx を起票して再確認
4. 既存実装への影響を FIX として起票

## コメントベースタスク管理

ソースコード内のコメントでタスク・修正・質問を管理する。外部ファイル（task.md 等）は使わない。

### タグ定義

| タグ        | 意味                     | 状態             | 誰が書く        |
| ----------- | ------------------------ | ---------------- | --------------- |
| `TODO :`    | 実装予定・検討事項       | 未着手           | ユーザー / AI   |
| `@fix`      | 修正指示（バグ、不整合） | AI対応待ち       | ユーザー / Opus |
| `@question` | 確認・相談・質問         | 回答待ち         | ユーザー / AI   |
| `@done`     | 修正完了                 | ユーザー確認待ち | Codex / Sonnet  |

### @question の使い方拡張

`@question` は2種類に使い分ける。

#### 一般的な質問
通常の確認・相談はそのまま `@question` で記述。

#### 仕様未確定の確認(CONFIRM 相当)
仕様書(docs/)に記載がない事項を確認する場合、A/B/C 形式で選択肢を提示する。
規模が大きい・複数箇所に影響する場合は CONFIRM チケット化する(下記参照)。

例(コメントで完結する軽量版):
```
// @question-spec: エラー時のレスポンス形式が docs/ に未記載
//   A. ['status' => 'error', 'message' => $msg] (現状)
//   B. WP_Error オブジェクト
//   C. その他
```

ユーザー回答後の処理:
1. 必要に応じて docs/ の該当ファイルを更新
2. @question-spec を削除
3. 仕様変更による修正が必要なら @fix を新規追加

#### CONFIRM チケット化が必要な場合
以下に該当する場合、docs/tickets/CONFIRM-xxx.md を起票する:
- 複数箇所に影響する仕様判断
- 設計方針レベルの判断が必要
- ユーザー回答後に複数の FIX が派生する可能性

### ライフサイクル

```
TODO : 実装予定のメモ
  ↓ 実装方針が決まったら
@fix: 具体的な修正指示
  ↓ Codex/Sonnet が修正・報告
@done: 修正内容の要約
  ↓ Opus がソースコードをレビュー（必須）
✅ OK → ユーザーに報告・コメント削除可
❌ NG → @fix に戻して再指示
  ↓ ユーザー確認OK
（コメント削除）
```

`TODO :` はユーザーが自分で実装する場合もある。その場合は直接削除して OK。

### Opus レビュー（@done 受領時・必須）

Sonnet/Codex から完了報告を受けたら、Opus は必ず以下を確認する：

1. **変更ファイルを読む** — 報告にあるファイルをすべて Read する
2. **実装チェック**
   - @fix の仕様を満たしているか
   - 周辺コードに意図しない変更がないか
   - `value/page/*.php` との整合性が取れているか
   - 構文チェック結果が正常か（php -l / node --check）
3. **結果を報告**
   - ✅ OK: 「レビューOK。@done コメント削除可」と報告
   - ❌ NG: 問題箇所を明示し、@done を @fix に戻して再指示
4. **仕様書との同期確認(必須)**
   - 実装が docs/ の仕様書と一致しているか
   - 仕様変更があった場合、仕様書も更新されているか
   - 未更新の場合、Opus が docs/ の該当ファイルを自動更新する
   - 更新内容はチャットで「仕様書 docs/xxx.md を以下のように更新: [差分]」と報告
   - ユーザーが内容確認後、コミットに含める
   - 仕様書未更新の状態で @done コメント削除を許可しない

**完了判定の正本**:
レビュー OK の判定基準は「## チケット管理 > ### 実装完了の基準(厳格化)」を参照。
Opus は本セクションのチェックを行い、最終的な完了可否は「実装完了の基準」5項目すべてを満たすかで判定する。

### レビューチェックリストの遵守

レビュー実施時は必ず docs/checklist-review.md を参照し、
担当セクション(Opus / Codex)を上から順に全項目確認する。

スキップ禁止。観点の漏れを防ぎ、レビュー品質を安定させる。

#### Opus レビュー時の追加ステップ

既存レビュープロセス（上記 1-3）に以下を組み込む:

- **Step 0（物理確認の前）: docs/checklist-antipatterns.md の該当パターン確認**
  - 過去のハマりパターンに該当しないかチェック
  - 該当する場合は指摘 + `docs/checklist-antipatterns.md` の該当箇所を引用
- **物理確認**（ファイル参照を含む処理の場合）
  - `docs/checklist-diagnosis.md` 参照
- **原因分析の実施**
  - 症状 → 直接原因 → 根本原因を特定 → docs/tickets/ に記録
- 修正プロンプト出力前に、以上 3 点が完了していることを確認

### ダブルレビュー(TASK 完了時)

通常の FIX 対応は Opus 単独レビューで完結。
TASK 完了のタイミングで GPT(Codex)による反論レビューを実施。

#### 発動タイミング

##### 必須
- 大規模 TASK(複数ファイル、複数日かかる)の完了時
- 関連する複数 TASK の完了時(機能単位でグルーピングして1回)
- セキュリティ・認証・決済・DB スキーマ変更を含む TASK の完了時

##### 任意
- 小規模 TASK(1〜2ファイル、数時間)の完了時はスキップ可
- ユーザー判断で「これは念のためダブルチェックしたい」場合は実施

#### グルーピングの判断基準
以下に該当する複数 TASK は、まとめて1回のダブルレビューで対応:
- 同じ機能領域(認証・決済・検索 等)
- 同じ画面・コンポーネント
- 同じ DB テーブル操作
- 同じリリースに含める予定

#### フロー
1. 対象 TASK の Opus 単独レビューが全て完了
2. あさこが「TASK-xxx, TASK-yyy のダブルレビュー実施」と Opus に依頼
3. Opus: 横断的レビュー(設計・仕様整合・横展開・グループ全体の一貫性)
4. ユーザーが Opus のレビュー結果を GPT(Codex) にコピペ
5. GPT: 反論レビュー(実装詳細・型・NULL処理・既存コード副作用)
6. ユーザーが GPT 追加指摘を Opus に貼る
7. Opus が docs/tickets/ と docs/progress.md に追加 FIX を反映
8. REVIEW-xxx チケットを起票し、レビュー結果(件数・派生チケット・所感)を記録

観点を分離し、同じ観点で2回見ない。

#### 横断レビューの観点
複数 TASK をまとめてレビューする時は、単独レビューでは見えない以下を重点確認:
- TASK 間の整合性(命名・パターン・データ構造)
- 共通処理の重複・抜け漏れ
- 機能単位での網羅性
- TASK 完了後に派生した FIX の総数・傾向

### コメント例

```php
// TODO : sidenavにスクロール追従のstickyを追加
↓ AI に依頼する場合
// @fix: sidenavにposition: stickyを追加。_blocks.scssのb-sidenavセクション
↓ 修正後
// @done: b-sidenavにsticky追加済み
↓ 確認OK後
// （コメント削除）
```

### 検出コマンド

```bash
# タスク一覧（TODO + 未対応の@fix + 質問）
grep -rn "TODO :\|@fix\|@question" _src/

# AI対応待ちのみ
grep -rn "@fix\|@question" _src/

# 確認待ち
grep -rn "@done" _src/
```

### Codex/Sonnet への修正完了ルール

- **報告の冒頭にタスクID を必ず明記すること**（例: `ACF-1A-バッチ2 完了`）
- 修正完了後、タグを `@done` に変更し修正内容を追記
- コメントは削除しない（Opus レビュー後に削除）
- **構文チェックを必ず実行し、結果を報告に含める**
  - PHP: `php -l ファイル名`
  - JS: `node --check ファイル名`
  - SCSS: `npm run build:vite` で確認
  - 構文エラーがある場合は `@done` にせず修正する
- **報告は下記フォーマットの1つのコードブロックで出力すること（ユーザーが Opus にコピーしやすい形）**

```
[タスクID] 完了

変更ファイル:
  - ファイルパス:行番号 — 変更内容

構文チェック:
  - php -l ファイルパス → エラーなし

@done 箇所:
  - ファイルパス:行番号 // @done: 内容

受け入れ条件の確認:
  - [✓] 条件1: 実行結果(ログ / スクリーンショット / 出力例)
  - [✓] 条件2: 実行結果
  - [-] 条件3: (未確認の理由)

確認方法の詳細:
  - (どう実行して、どう動いたか)

未対応・懸念点:
  - なし / あれば記載

Opus: レビューをお願いします
```

「動きました」だけの報告を禁止。何をどう確認したかを必ず記録。

## チケット管理(docs/tickets/)

### 種別と用途

| ID 接頭語    | 用途                          | 動作変更 |
| ------------ | ----------------------------- | -------- |
| TASK-xxx     | 実装タスク(新機能・機能改善)  | あり     |
| FIX-xxx      | バグ修正                      | あり     |
| REFACTOR-xxx | リファクタリング(コード整理)  | なし     |
| CONFIRM-xxx  | 仕様確認(ユーザー判断待ち)    | -        |
| REVIEW-xxx   | レビュー記録(TASK完了時)      | -        |

### ファイル命名

`docs/tickets/<TYPE>-<番号>-<短い英語の概要>.md`

例:
- docs/tickets/TASK-001-error-judgment.md
- docs/tickets/FIX-002-type-mismatch.md
- docs/tickets/REFACTOR-001-auth-service.md
- docs/tickets/CONFIRM-001-error-format.md
- docs/tickets/REVIEW-001-auth-task-group.md

### テンプレート

- docs/tickets/TEMPLATE-TASK.md
- docs/tickets/TEMPLATE-FIX.md
- docs/tickets/TEMPLATE-REFACTOR.md
- docs/tickets/TEMPLATE-CONFIRM.md
- docs/tickets/TEMPLATE-REVIEW.md

### 書き込み権限

- Opus: 全チケットの作成・更新
- Codex/Sonnet: 読み取り専用、完了報告はチャット内
- ユーザー: 必要に応じて編集可

### @fix(コメント) と チケット の使い分け

#### @fix(インラインコメント)
- 1ファイル内で完結する修正
- 仕様に影響しない軽微な修正
- タイポ・コメント修正等の自明な修正
- 数行の変更で終わる

#### docs/tickets/FIX-xxx.md (ファイル管理)
- 複数ファイルにまたがる修正
- 仕様との整合性検討が必要
- 原因分析が必要
- 再発防止価値がある
- 再修正が発生(2回以上)
- セキュリティ・データ整合性に関わる

#### 昇格
@fix で始めた修正が複雑化した場合、docs/tickets/FIX-xxx.md に昇格:
1. docs/tickets/FIX-xxx.md を TEMPLATE-FIX.md から作成
2. @fix コメントを `@fix FIX-xxx: タイトル` 形式に更新
3. docs/tickets/ に原因分析と修正履歴を記録

### REFACTOR の扱い

- 動作を変えない修正のみ
- FIX(動作変更あり) と混ぜない
- レビュー時は「機能が壊れていないか」を重点確認

### 大規模 FIX / TASK の対応(影響範囲マップ + ブロック分割)

複数ファイル・複数機能に波及する FIX / TASK は、以下の手順で対応する。

#### Step 1: 影響範囲マップ作成(着手前・必須)

Opus が grep / コード解析で影響範囲を列挙し、tickets/ に記録:

```markdown
## 影響範囲マップ

### 直接変更
- foo.php(L42-88)
- bar.php(L15-30)

### 呼び出し元(回帰確認対象)
- baz.php(foo() を呼ぶ箇所)
- qux.php(bar() を呼ぶ箇所)

### 関連データ
- value/page/xxx.php(該当キー)

### 関連スタイル
- _xxx.scss(該当セレクタ)

### 関連ドキュメント
- docs/xxx.md
```

マップなしで実装に着手することを禁止。

#### Step 2: ブロック分割レビュー(完了後・必須)

影響範囲マップを 5〜10 のブロックに分割し、1ブロックずつレビュー:

```
ブロック1: 直接変更ファイル(critical / 仕様整合)
ブロック2: 呼び出し元(回帰観点)
ブロック3: データ整合性(value/page との照合)
ブロック4: スタイル整合性(SCSS との照合)
ブロック5: ドキュメント整合性(docs/ 更新確認)
```

各ブロック完了で「critical ゼロ」を確定してから次へ。
全ブロック完了後、横断レビューで「ブロック間の境界問題」を確認。

#### 対象判断基準

以下に該当する FIX / TASK は本フローを適用:
- 3ファイル以上に変更が及ぶ
- 共通関数 / setHtml* / value 構造の変更を伴う
- DB 構造変更を伴う
- セキュリティ / 認証 / 権限処理の変更

### 受け入れ条件(TASK起票時・必須)

TASK起票時に「これが満たされたら完了」を明記する。
docs/tickets/TEMPLATE-TASK.md の「要件」セクションを以下の形式で記入:

```markdown
## 受け入れ条件

### 機能
- [ ] 条件1: (具体的な動作)
- [ ] 条件2: (具体的な動作)

### エッジケース
- [ ] NULL / 空データでの挙動
- [ ] 最大データ量での挙動
- [ ] 権限なしユーザーでの挙動

### 表示(UI)
- [ ] SP / TB / PC で表示崩れなし
- [ ] 全パターン(N種類)の表示確認

### データ整合性
- [ ] DB 保存内容が想定通り
- [ ] value/page との一致
```

AI実装後、各条件を1つずつ潰して報告する。

### 実装完了の基準(厳格化)

「実装完了」と「テストシナリオ実施可能」は同義とする。
以下を満たさない限り、TASK / FIX を完了状態にしない。

#### 必須クリア項目

1. **受け入れ条件の全クリア**
   - 「受け入れ条件」セクションの全チェックボックスが ✓ になっている
   - 未確認項目が1つでもあれば未完了扱い

2. **動作確認の証拠提示**
   - 各受け入れ条件に対して、実行ログ / スクリーンショット / 出力例を添付
   - 「動きました」だけの報告は無効

3. **エッジケース確認**
   - NULL / 空 / 最大値 / 権限なし の最低4パターンを確認済み
   - 該当なしの場合は「該当なし(理由)」と明記

4. **Opus レビュー OK**
   - critical / 要修正 ともにゼロ
   - 軽微のみ残存は完了可

5. **仕様書同期完了**
   - 実装に伴う仕様変更があれば docs/ 更新済み

#### 完了判定フロー

```
Sonnet/Codex 実装完了報告
  ↓
受け入れ条件チェック → 全クリア?
  ├─ No → 未完了として差し戻し(具体的に何が未確認か明示)
  └─ Yes ↓
動作確認証拠の提示 → あり?
  ├─ No → 差し戻し(証拠提示を要求)
  └─ Yes ↓
Opus レビュー → critical / 要修正 ゼロ?
  ├─ No → FIX 起票
  └─ Yes ↓
仕様書同期 → 完了?
  ├─ No → docs/ 更新を要求
  └─ Yes ↓
✅ 実装完了 → テストシナリオ実施可能
```

#### テストシナリオの位置づけ

- テストシナリオは「仕様通り動くか」の最終確認
- critical / 要修正レベルのバグが発見される状態は「実装未完了」とみなす
- シナリオ中に critical が出たら、シナリオを中断して FIX 対応 → 再度「実装完了基準」を満たしてから再開

#### 禁止事項

- 受け入れ条件未定義のまま実装に着手すること
- 動作確認証拠なしで「完了」と報告すること
- 軽微のバグを残したままシナリオに突入することは可、ただし critical / 要修正は不可
- 「とりあえずシナリオ回して見つかったら直す」運用を禁止

### 仕様意図のズレ防止

「それっぽい実装」(仕様の意図と微妙にズレた実装)を防ぐため、以下を必須化。

#### 実装前: 解釈確認

Sonnet/Codex は実装着手前に、仕様の解釈をユーザーに確認する。

```markdown
## 仕様解釈確認

### 読んだ仕様
- docs/xxx.md(該当セクション)

### 解釈した内容
1. (この機能はこういう動作をするものと理解)
2. (この値はこう判定すると理解)
3. (このパターンは N 種類あると理解)

### 不明点・選択肢が分かれる箇所
- A. (解釈案A)
- B. (解釈案B)

実装前に確認をお願いします。
```

ユーザー承認後に実装着手。

#### 実装後: 仕様意図の対応表

レビュー時、Opus は「仕様意図 ↔ 実装」の対応表を作成。

```markdown
## 仕様意図 ↔ 実装対応表

| 仕様(docs/xxx.md) | 実装(file:line) | 一致 |
| --- | --- | --- |
| 受験上限は動的判定 | foo.php:L42(固定値で実装) | ❌ |
| カード表示は7パターン全対応 | bar.php:L88(2パターンのみ) | ❌ |
| エラー時は WP_Error | baz.php:L15 | ✅ |
```

不一致(❌)があれば critical 指摘として FIX 起票。
固定値や限定パターン実装が、仕様の意図に沿っているか必ず確認。

### CONFIRM の扱い

- 仕様書に記載がない事項のユーザー確認用
- A/B/C 形式で選択肢を提示
- ユーザー回答後、必要なら docs/ を更新
- 派生する修正は別 FIX として起票

### 派生チケットの紐付け

チケット作成時、関連する親チケットを必ず記載。
親チケットの「派生したチケット」セクションにも追記。

### クローズ手順

1. docs/tickets/ の「クローズサマリ」を記入
2. docs/progress.md のステータスを「完了」に更新
3. @done コメントの削除
4. docs/checklist-antipatterns.md への追記判断と実施

### checklist-antipatterns.md

- **配置**: `docs/checklist-antipatterns.md`
- レビュー時に Opus が参照、該当パターンがあれば指摘
- 解決した FIX で再発防止価値があるものは追記
- 新規実装時の参考資料として使用

## docs/progress.md(チケット俯瞰)

全チケットの進捗を俯瞰する一覧。詳細は docs/tickets/ 各ファイル参照。

### 規模別2モード

TASK の規模に応じて、通常モードと大規模モードを使い分ける。

#### モード判定基準

「### 大規模 FIX / TASK の対応」の判定基準を流用:
- 3ファイル以上に変更が及ぶ
- 共通関数 / setHtml* / value 構造の変更を伴う
- DB 構造変更を伴う
- セキュリティ / 認証 / 権限処理の変更

該当なし → 通常 TASK モード
該当あり → 大規模 TASK モード(フェーズ表示)

### 通常 TASK モード(デフォルト)

```markdown
### TASK-xxx: タイトル
ステータス: 実装中 / レビュー / テストシナリオ / 完了
優先度: 高 / 中 / 低

- [ ] 実装
- [ ] レビュー(Opus)
- [ ] テストシナリオ(該当時のみ。不要なら「該当なし」と明記)
- [ ] 派生 FIX
  - [ ] FIX-xxx: タイトル → 状態
- [ ] 完了確認(受け入れ条件 + 動作確認証拠)
```

### 大規模 TASK モード(複数日 / 3ファイル以上 / セキュリティ系)

```markdown
### TASK-xxx: タイトル
ステータス: フェーズN進行中
優先度: 高

#### フェーズ1: 設計(モック / データ / function 各CP)
- [ ] 画面モック作成
- [ ] データ枠定義
- [ ] function 設計
- [ ] CONFIRM-xxx の解決(あれば)

#### フェーズ2: 実装
- [ ] 仕様解釈確認
- [ ] 実装
- [ ] 動作確認証拠提示

#### フェーズ3: 単独レビュー & FIX
- [ ] Opus レビュー
- [ ] FIX 対応
- [ ] 再レビュー OK

#### フェーズ4: ダブルレビュー & FIX(任意・大規模時のみ)
- [ ] GPT 反論レビュー
- [ ] FIX 対応
- [ ] REVIEW-xxx 起票

#### フェーズ5: テストシナリオ
- [ ] シナリオ実施
- [ ] テスト中検出 FIX 対応(あれば)
- [ ] 完了確認
```

### 全体構造

```markdown
# 進捗管理

## アクティブ
```

## Git 運用

### 基本方針

- AI による変更は必ずコミット単位で追跡可能にする
- 失敗時は `git revert` で安全にロールバック可能にする

### コミット粒度

- 1 タスク（TASK / FIX / REFACTOR-xxx）= 1 コミット を原則
- リファクタリングと機能変更は別コミット
- フォーマット調整と機能変更は別コミット

### コミットメッセージ規約

フォーマット:

```
[種別] タスクID: 概要
```

**種別**:

- `feat`: 新機能（TASK-xxx）
- `fix`: バグ修正（FIX-xxx）
- `refactor`: リファクタリング（REFACTOR-xxx）
- `docs`: ドキュメント
- `chore`: ビルド・設定

**例**:

```
[fix] FIX-002: 型不整合を is_numeric 検証に修正
```

### AI への指示

- **AI は直接 commit しない**
- タスク完了時、推奨コミットメッセージとファイル一覧を提示
- ユーザーが内容確認後に commit 実行

### ロールバック

- `git revert HEAD` で直前を取り消し
- `git log --grep="FIX-xxx"` で特定 FIX を検索
- `git checkout -- <ファイルパス>` でコミット前の変更を破棄

## Opusへの@fix相談フロー

### 概要

「問題はわかるが、どこに `@fix` を置くべきかわからない」場合、
Opus に相談 → Opus が該当箇所を特定してソースに `@fix` を直接追記 → Codex/Sonnet 向けプロンプトを出力する。

### プロンプト例

```
〇〇の処理で△△が起きるんだけど、どこに@fixを置けばいい？
```

### フロー

1. **ユーザーが Opus に相談**（問題の症状・期待する動作を伝える）

2. **Opus が調査・特定**
   - `value/page/*.php` のデータ定義と実装コードを照合
   - 原因箇所を特定（複数の場合あり）

3. **Opus がユーザーに報告**

```
## @fix 配置提案

### 問題
[ユーザーの報告内容を要約]

### 原因箇所
1. ファイル: [パス]
   行: [行番号付近]
   理由: [なぜここが原因か]
   @fix内容: [追記するコメントの案]

2. （複数ある場合）

### 担当判断
[Codex / Sonnet — 理由: 既存コード修正のため / 新規実装が必要なため]

追記してよいですか？
```

4. **ユーザーが承認**

5. **Opus がソースに @fix コメントを直接追記**
   - 追記するのはコメントのみ（ロジックの変更は一切しない）

6. **Opus が担当別プロンプトを出力**
   - Opus が修正内容から Codex（既存コード修正・差分対応）か Sonnet（新規実装が必要）かを判断
   - 担当名を明記し、コピー対象部分はコードブロックで出力する

#### Sonnet / Codex 向け指示プロンプトのフォーマット

- 指示全体（説明・コード・完了条件・担当）を **1つのコードブロック** に収める
- ブロック内にコードが含まれる場合は、コードブロックを使わず **インデントで表現** する
- 1回のコピー＆ペーストで Sonnet / Codex に渡せる状態にする

プロンプト出力時は、**タイトル・説明文・修正指示のすべてを1つのコードブロック内に収める**。
VSCode 上でコードブロックのコピーボタンからワンクリックでコピーできるようにする。
コードブロックの外に説明文を書かない。

````
🔧 [Codex / Sonnet] 修正指示
以下をコピーして [Codex / Sonnet] に渡してください。

```
対象:
- ファイル: [パス] 行XX
- コメント: // @fix: [内容]

期待する修正内容:
[具体的に何をどう直すか]

修正範囲:
[このファイルのみ / 他にも影響あれば列挙]

完了条件:
- [確認項目]
- 修正後 @fix を @done に変更し、修正内容を追記

担当: [Codex / Sonnet]
理由: [既存コードの修正のため / 新規実装が必要なため]
```
````

- 指示プロンプトの見出しに **タスクID**（例: F3, R5b）を含めること
  - 形式: `🔧 [Codex / F3] 修正指示` または `🔧 [Sonnet / バッチ1-M] 実装指示`
- Codex / Sonnet は報告時にも **タスクID** を明記すること
  - 形式: `F3 完了` `バッチ1-M 完了`
    ※ 複数箇所ある場合は、担当ごとにコードブロックを分けて出力する。

### 注意

- Opus がソースに追記するのは `TODO :` / `@fix` / `@question` コメントのみ
- ロジックの変更・追加は一切しない
- ユーザー承認なしに追記しない
- 報告 → 承認 → 追記 → プロンプト出力まで1回の流れで完結させる
- Codex か Sonnet かは Opus が判断し、プロンプト内に必ず明記する

## CLAUDE.md の編集ルール

- CLAUDE.md の変更は **ユーザーの許可を得てから** 行う
- AI が勝手に追記・変更することは禁止
- タスク管理はソースコード内のコメント（`TODO :` / `@fix`）で行う
- 新しい `setHtml*` 関数を作成した場合、コンポーネント一覧への追記も行う（Opus が担当）

## 質問・不明点がある場合

1. 実装を止める
2. 具体的に何がわからないか説明する
3. ユーザーの回答を待つ
4. 回答を得てから作業を再開する

**「たぶんこうだろう」で進めない**
