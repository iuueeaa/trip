## Opus 担当(設計・仕様・横断的観点)

### 仕様整合性
- [ ] docs/ の仕様書と実装が一致しているか
- [ ] 仕様書未記載の事項を推測で実装していないか(CONFIRM化すべきか)

### 設計の一貫性
- [ ] 同じロジックが複数箇所に重複していないか
- [ ] 判定方式の統一(例: DB判定 vs ロジック判定の混在がないか)
- [ ] 命名規則の一貫性(関数名・変数名・クラス名)
- [ ] 責務分離(1つの関数が複数の責務を持っていないか)

### カバレッジ(設計時・レビュー時の両方で確認)

#### 仕様カバレッジ
- [ ] docs/ の仕様書に記載された全要件が実装されているか
- [ ] 仕様書の「要件」「受け入れ条件」のリストを1項目ずつ実装と照合
- [ ] 未実装の要件があれば、意図的(別TASK)か漏れか明確にする
- [ ] 仕様書にない実装が含まれていないか(スコープ外の機能追加)

#### テストカバレッジ
- [ ] 主要な処理パスが動作確認されているか
  - 正常系: 期待通りの入力での動作
  - 異常系: NULL / 空 / 不正値 / 権限なし
  - 境界値: 最小 / 最大 / 境界前後
- [ ] エラーハンドリング分岐が全て確認されているか
- [ ] 条件分岐(if/switch)の全ブランチが通るか
- [ ] ループ処理が 0件 / 1件 / 複数件で動作するか

#### UI/画面カバレッジ(WEB制作時)
- [ ] 全ページ種別(archive / single / page)で表示確認
- [ ] 空データ時の表示
- [ ] 最大データ時の表示崩れ
- [ ] レスポンシブ(SP / TB / PC)確認

#### 実施タイミング
- 設計時: 仕様カバレッジを確認し、TASK の要件リストに全項目を含める
- レビュー時: 設計時の要件リストと実装を照合、テストカバレッジも確認
- カバレッジ漏れが見つかったら FIX または別 TASK として起票

### 横断的観点
- [ ] 指摘したバグパターンが他箇所にも存在しないか(横展開検索)
- [ ] 親 TASK の派生 FIX に傾向がないか
- [ ] docs/checklist-antipatterns.md の既出パターンに該当しないか

### value/page 整合性(WEB制作基本形)
- [ ] 静的データのキー名と ACF フィールド名が一致
- [ ] PHPテンプレートの参照キーと value 定義が一致
- [ ] CSS クラスが p- 接頭語で統一

## Codex 担当(実装・ローカル観点)

### データ処理
- [ ] NULL / undefined の処理
- [ ] 空配列・空文字の処理
- [ ] 型の一貫性(int/string 境界)
- [ ] 配列キーの存在チェック

### セキュリティ
- [ ] SQLインジェクション対策(prepared statement)
- [ ] XSS対策(esc_html / esc_attr)
- [ ] nonce検証(WordPress)
- [ ] 権限チェック(current_user_can)

### エラー処理
- [ ] try-catch の適切な使用(対症療法的な覆い隠しでないか)
- [ ] エラーメッセージの妥当性
- [ ] ログ出力の有無

### 既存コードへの影響
- [ ] 変更した関数の呼び出し元すべてに影響確認
- [ ] 既存の動作を壊していないか(回帰)
- [ ] 不要なコメント削除・変数名変更がないか

### 構文
- [ ] php -l / node --check でエラーなし
- [ ] ビルドエラーなし
