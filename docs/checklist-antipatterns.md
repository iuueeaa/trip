# アンチパターン集 (antipatterns)

過去にハマったパターンを蓄積し、再発を防ぐ。
新規実装・レビュー時は本ファイルを参照する。

## 運用ルール
- 解決した FIX のうち「次も同じハマり方をしそう」なものを追記
- Opus はレビュー時に本ファイルを参照し、該当パターンがあれば指摘

## テンプレート

### YYYY-MM-DD: タイトル
- 何が起きたか: (症状)
- 原因: (根本原因)
- 対策: (今後の対処方針)
- 該当 FIX: FIX-xxx (あれば)
- 検出方法: (どうやれば早く気付けるか)

### 2026-04-21: Sonnet の「厳密等価」逸脱傾向
- 何が起きたか: MOD-3 初回で `setAcfDtDdRepeater` への移行時、元コードの
  `'width' => 30` と `'placeholder'` オプションを「ヘルパーのオプションにないから」
  と省略。DB 互換性は維持されたが ACF 管理画面 UI が変化した
- 原因: Sonnet は「ヘルパー設計を優先し、呼び出し側の細部を省略する」判断を取りがち
- 対策:
  - 指示書の完了条件に「出力配列が書き換え前と**厳密に等価**」を必ず明記
  - ヘルパー関数の options は「呼び出し元が必要とする全ての属性」をサポートする設計にする
  - Sonnet の報告で「省略した」「スキップした」等の表現があればレビューで重点確認
- 該当 FIX: なし（MOD-3b で追補対応）
- 検出方法: ヘルパー関数を新設する際、呼び出し側の既存コードと生成結果を配列キーレベルで一致確認

### 2026-04-22: 使用中コンポーネント削除時の 5 層対応の必要性
- 何が起きたか: MOD-8 で gallery コンポーネント完全削除時、「ファイル自体削除」だけでは不十分で、5 層の対応が必要だった（呼び出し 4 箇所 / value 定義 3 ファイル / SCSS 3 箇所 / docs 2 箇所 / 本体 1 ファイル）
- 原因: Opus の初回判定（段階3 レビュー）で「gallery.php 内の大規模コメントアウト削除」を推奨したが、ユーザーは「本体削除」を選択。その時点で関連する 5 層（PHP 呼び出し / value 層 / SCSS / docs / 本体）の影響範囲調査が必須になった
- 対策:
  - 「使用中コンポーネント」の削除判断を AI が出す前に、以下 5 層の影響を必ず列挙:
    1. PHP 呼び出し箇所（page / template / block 等）
    2. value 層の対応キー（value/page/*.php の `'xxx' =>` 定義）
    3. SCSS 層（global / layout 層の `.p-xxx` / `.b-xxx` 定義）
    4. docs 層（CLAUDE.md / docs/*.md の一覧・参照）
    5. 本体ファイル（function/parts/*.php）
  - 削除実行は「呼び出し → value → SCSS」の Codex 指示書、「docs → 本体」の Opus 作業の 2 段階で進める
  - docs/php-to-shopify-manual.md など別コンテキストの md は対象外として明示
- 該当 FIX: MOD-8（ミクロレビュー段階3 の一部）
- 検出方法: 削除判断を出す前に 5 層の grep を実施し、影響範囲を報告に含める

### 2026-04-21: Opus の指示書内ファイル実在チェック漏れ

  - 指示書の対象ファイルは grep 結果から抽出する（推測で追加しない）
- 該当 FIX: なし

### 2026-04-27: 「コードは動くがデータが伴っていない」状態のレビュー漏れ
- 何が起きたか: ロジック実装は仕様通り、自動テストも green。しかし実機テストで「想定するデータが存在しない」ことで実際は動作しない
- 原因:
  - レビュー観点が「コードの正しさ」に偏り「データセットアップ前提の検証」が抜ける
  - FIX 起票時に「実装」と「データ追加」の TASK を別々にしないとデータが取り残される
  - テストシナリオが「実機 + 必要データ揃った状態」を暗黙の前提にしている
- 対策:
  - FIX 起票時に「必要なデータ(migration_data.json / シード)」を明示
  - 実装 FIX と「データ整備 TASK」をセットで起票
  - レビュー時に「このコードが必要とするデータが migration data 等に存在するか」を grep で確認
  - テストシナリオに「前提データ条件」セクションを追加
  - E2E テストで実 DB 状態を検証する仕組みを段階的に導入
- 該当 FIX: なし
- 検出方法: レビュー時に該当コードが参照するデータキーを grep し、シード/migration への存在を確認
